## 작업 내용

### 신규 — Telegram.js
- `tgSendMessage` / `_tgFormatPnL` (대시보드 동일 수치 — `trendTotalProfit`·`totalProfitRate`·`dayChangAmount`·`dayChangePct`·`priceAsOfDate`)

## 주요 결정

1. **메신저 우회 구조** — 워치엔 인증 정보 없음. 시크릿은 GAS에만. 양방향(수동 갱신) + 자동 푸시 둘 다 지원
2. **자동 푸시 시간대** — 거래일 09:00~16:00만 (`_mIsTradingDay()` + 분 단위 cutoff `mins >= 540 && mins <= 960`)
3. **트리거 구조** — GAS `everyMinutes(20)` 미지원이라 `everyHours(1).nearMinute(0/20/40)` 3개로 분리
4. **수동 갱신 경량화** — `newMobileUpdateAll()` 너무 무거워 30초+ → Telegram retry 무한 루프 발생. 대신 `updateNewPriceHistory` + `updatePositionFromLedger`만 (대시보드 렌더·추이 생략)
5. **webhook 인증 이중 검증** — secret(URL query) + chat_id 화이트리스트
6. **Web App "Anyone" 액세스** 필요 — Telegram이 익명 POST. URL 노출돼도 secret 없으면 forbidden

## 배포
- GAS: `python3 apps-script-v2/push_safe.py` — Telegram.js·Main.js 등 10개 push (여러 차례)
- Web App 새 배포 (Anyone 액세스) — Telegram webhook용
- **deployment 버전 갱신 1회** — push로 head 코드는 갱신되지만 Web App은 deployment 시점 코드 고정. 무한 루프 수정 코드 반영 위해 "배포 관리 → 신규 버전" 필요했음

## 발견된 버그 (해결됨)

### 무한 루프 — Telegram webhook retry × 무거운 갱신
- **증상**: "갱신" 1번 보냈는데 봇이 메시지 보내고 또 "갱신중..." 반복 무한
- **원인**: `tgRefreshAndPush`가 `newMobileUpdateAll()` 호출 → 30초+ 소요 → Telegram이 webhook 응답 못 받았다고 판단 → 같은 update_id로 retry → 또 처리 → 무한 루프
- **해결**:
  1. `update_id` 중복 제거 (`PropertiesService`에 마지막 처리한 ID 저장)
  2. `LockService.tryLock(1000)` — 동시 처리 차단
  3. 갱신 작업 경량화 (대시보드 렌더·추이 생략)
  4. `setWebhook` 옵션에 `max_connections:1` 추가
- **별도 함정**: Web App deployment는 배포 시점 코드를 영구히 실행함. `clasp push`만으론 반영 안 됨 → "배포 관리 → 신규 버전" 필수
