# 설계 노트 — 민감 파일 private 이전 + public 히스토리 재작성

2026-07-04. 사용자 승인: 옵션 A(git-filter-repo) + 투자논리/프롬프트(보유 종목 목록 포함) private 이전 + 히스토리 일괄 제거.

## 범위
- **private 이전 (FD5to6-reports `_config/`)**: `us-prompt.md`·`kr-prompt.md`·`weekly-prompt.md`·`kr-theses.md` (보유 종목 16개 목록·투자 논리). `send_telegram.py`는 비민감 — main 유지.
- **히스토리 purge 경로**: `docs/reports/` + 위 4개 파일 (전 히스토리).
- **잔존 수용(명시)**: 세션 문서·pending·errors에 흩어진 종목명·수익률 % 단편은 이번 범위 밖(전량 스크럽은 비현실적, 민감도 낮음). 필요 시 별도 결정.

## 1. 외부 동작 가정
- 워크플로 Generate가 `cat docs/reports/_config/<prompt>.md` — reports checkout이 Generate보다 앞이라 파일 존재(사실, yml 순서 확인). 참조 지점 전수: yml 3곳 + 프롬프트 내부 Read 경로 2곳(kr-prompt:42·weekly-prompt:29) — grep으로 확정.
- git-filter-repo가 지정 경로를 전 커밋에서 제거, 나머지 커밋은 SHA만 변경(표준 동작). 빈 커밋([skip ci] 리포트 커밋들)은 자동 drop.
- force push 후 GitHub에 dangling 객체가 일시 잔존 가능(SHA 알아야 접근) → Support gc 요청문 제공으로 보완.
- 다른 맥/CI: 이후 checkout은 새 SHA 기준 — in-flight run 무영향. 다른 맥은 재clone 필요(폴더 삭제 후 실행@).

## 2. 과거 부류 (errors.md)
- "push 실패 가림"(6/10) → force push도 파이프 없이 exit code 확인 + `git ls-remote`로 원격 HEAD 대조.
- "배포↔push 시각" 부류 → 재작성 직후 dry_run 1회로 워크플로 살아있음 재확인.

## 3. 추론 가능 vs 실환경 전용
- 머리로 거름: 경로 참조 전수 목록, 커밋 순서(이전 커밋 먼저 → 재작성이 그 삭제 커밋까지 정리), 로컬 재정렬(fetch+reset --hard, untracked인 backups/·docs/reports clone 무영향).
- 실환경에서만: filter-repo 결과 검증(fresh clone에서 `git log --all -- <경로>` = 0), 재작성 후 dry_run 정상.

## 4. 검증
- 자동: ① 이전 후 dry_run(KR) — private _config 프롬프트로 생성 성공 ② 재작성 후 fresh clone에서 purge 경로 log 0건 + `git log --all -S "kr-theses"` 파일단위 0 ③ 재작성 후 dry_run 재확인 ④ `git ls-remote origin main` = 로컬 HEAD.
- 사용자: 다른 맥 최초 사용 시 폴더 삭제→실행@ (재clone). GitHub Support gc 요청은 선택(요청문 제공).
