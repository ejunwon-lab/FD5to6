## KRX 직접 연동을 접은 이유 (이번 단계)

- KRX 정보데이터시스템은 비공식 form endpoint — 스키마·차단 정책 변동 리스크 + GAS(구글 US IP) egress 실측 불가(배포 전 검증 수단 없음). errors.md 2026-05-31 "PoP IP 차단" 부류.
- KRX 전용 휴장(연말 12/31·근로자의날)은 기존 **고정휴장 루프가 이미 커버** — KRX가 추가로 주는 것은 사실상 없음.
- 필요해지면 '출처' 컬럼 구조상 KRX를 제3 소스로 얹는 것 호환 (연 1회 교차검증 후보로 백로그).

## 변경 내용 (Holidays.js 2곳)

1. `syncHolidays()` — 채택 기준을 `title이 HOLIDAY_NAMES 매칭` → **`ev.getDescription()`이 '공휴일'로 시작**으로 교체.
   - **desc가 비면(구 API·변동 대비) 기존 화이트리스트로 폴백** — 최악의 경우가 현행과 동일 = 회귀 불가능 설계.

2. `scheduledHolidaySync()` — 12월 게이트 제거 → **매월 25일 실행** (트리거는 이미 monthly). 연중 임시공휴일 신설·재지정을 ≤1개월 지연으로 자동 수용. 멱등(수동 행 보존 + 전체 재작성)이라 반복 무해.
   - 고정휴장 루프(5/1·12/31·제헌절)·수동 행 보존·fallback 목록은 무변경.

## 1. 외부 동작 가정 + 근거

- 구글 공휴일 캘린더 이벤트에 분류 DESCRIPTION 존재 → 공개 ics 실측 (위). 캘린더 구독 데이터와 ics는 동일 소스.
- GAS `CalendarEvent.getDescription()` 메서드 존재 → GAS 공식 API 표준 메서드.
- **[실환경 검증 필요]** GAS CalendarApp이 이 캘린더 이벤트에서 ics와 같은 DESCRIPTION 문자열을 주는가 → 직접 실행 불가. **회피**: desc 빈 값이면 화이트리스트 폴백 → 검증 실패해도 현행 동작 유지. 실검증은 동기화 1회 실행 후 시트 확인(아래 4).
- 트리거는 HEAD 코드 실행 → 웹앱 재배포 불필요 (변경 함수는 메뉴·트리거 경로만, `_isKoreanHoliday`/`_getHolidaySet` 무변경 = 배포 경로 무영향).

## 3. 추론 가능 vs 실환경 전용

- 머리로 거름: 폴백 로직·멱등성·수동 행 보존·정렬·트리거 게이트 제거 영향(월 1회 캘린더 read 추가 — 무시 가능).
- 실환경에서만: GAS가 주는 실제 desc 문자열 → **동기화 1회 실행 + 시트 실측**으로만 확정. 폴백 덕에 실패 모드는 "현행 유지".

## 4. 검증 방법

- 자동: `node --check`(push_safe 내장) + walk-through 3건(제헌절 채택·스승의날 배제·desc빈값 폴백) + push_safe 배포.
- 사용자(또는 7/25 04:00 트리거 자동): **📊 시트 → 🛠️ 유지보수 → 🗓️ 휴장일 동기화** 클릭 → *휴장일* 시트에 **2026-06-03 지방선거일**(신규 — 분류 필터의 증거)과 **2026-07-17 제헌절** 행 존재, **스승의날·어버이날 없음**. toast에 "채택 N(공휴일분류)·제외 M(기념일)" 표기.

→ 게이트: **통과** (실환경 항목은 폴백으로 무해화 + 시트 실측 경로 확보)
