# 설계 노트 — 텔레그램 푸시 "오늘 +0원" 간헐 발생: KIS carry-forward 단서 표기

2026-07-04. 증상: 장중 푸시의 *오늘* 손익이 가끔 정확히 +0원(다음 푸시엔 정상).

## 원인 (코드 역추적으로 확정)
- *오늘* = Σ 종목지표.todayPnl = (현재가_이력 오늘행 − 전일행) × 수량 (StockMetrics.js:129-133)
- `updateNewPriceHistory`(NewSystem.js:889-896): KIS 조회 실패 종목은 **전일가 carry-forward**. KIS가 일시 전면 실패(토큰 갱신 순간·KIS 서버 순단)하면 오늘행 = 전일행 복제 → **전 종목 change=0 → 오늘 +0원**.
- 경고는 Logger·toast에만(906-911) — 텔레그램 수신자는 알 수 없음. 6/10 "+0 합계" 버그(부분갱신 stale, errors.md)와 다른 경로 — 그 수정(전체갱신 전환)은 유효하고, 이번 건은 **carry-forward의 무단서 발송**.

## 1. 외부 동작 가정
- ScriptProperties는 updateNewPriceHistory(쓰기)와 _tgFormatPnL(읽기)가 같은 스크립트라 공유됨 → 기존 사용례(tg_lastPushEpoch·DB.PROP_*)로 사실.
- 키 `kis_carried_status`는 기존 키와 무충돌 → grep으로 확인(사실).
- 푸시 흐름상 updateAllNew(→updateNewPriceHistory) **후** _tgFormatPnL 호출 → property는 항상 이번 갱신 결과 (Telegram.js tgPushPnL 순서로 확인).

## 2. 과거 부류 (errors.md)
- 2026-06-10 "+0원/직전값": 교훈 "표시 필드의 실제 출처 시트 역추적" — 본 노트가 그 절차로 원인 확정.
- 무음 실패 부류(2026-07-03 알림 설계와 동일 철학): 실패를 숨기지 않고 수신 채널에 단서를 태운다.

## 3. 추론 가능 vs 실환경 전용
- 머리로 거름: property 쓰기/읽기 순서, 0건일 때도 setProperty(전일 플래그 자동 해제), 부분 실패(n<total) 표기.
- 실환경에서만: 실제 KIS 순단 재현은 불가(대기). 다음 발생 시 푸시에 ⚠️ 라인 도착으로 최종 확인.

## 4. 검증
- 자동: push_safe `node --check` + walk-through(아래) + 배포 3개 버전·시각 API 확인.
- walk-through: KIS 전면 실패 가정 → carried=16=total → property `{date:오늘, carried:16, total:16}` → 같은 tgPushPnL 흐름의 _tgFormatPnL이 date==오늘·carried>0 → "⚠️ KIS 시세 조회 전체 실패 — 직전가 기준" 라인 추가 → 사용자가 +0원을 미갱신 신호로 해석 가능. 정상일 땐 carried=0 저장 → 라인 없음(기존 메시지 불변).
- 사용자: 다음 +0원 발생 시 푸시 메시지에 ⚠️ 라인 동반 확인.
