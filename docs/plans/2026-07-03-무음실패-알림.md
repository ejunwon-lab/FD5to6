# 설계 노트 — 리포트 무음 실패 텔레그램 경고 (market-report + telegram-push)

2026-07-03. 전반 점검 발견 P1: 리포트 생성 실패 시 job이 초록 성공으로 끝나 아무도 모름 / 체인 dispatch 3회 실패 시 run만 빨갛고 능동 알림 없음.

## 1. 외부 동작 가정
- Telegram Bot API `sendMessage`(`-d chat_id --data-urlencode text`)로 짧은 경고 발송 가능 → 같은 토큰·chat이 send_telegram.py로 매일 발송 성공 중(사실)
- `TG_CHAT_IDS`는 콤마 구분 → send_telegram.py:10·82 확인(사실)
- schedule 이벤트에서 `inputs.dry_run`은 빈 값 → `inputs.dry_run != true` 참 → 알림 armed. dispatch dry_run=true일 때만 억제(테스트 소음 방지) → GH 표현식 규칙(사실)
- 알림 step에서 `exit 1` → run 빨간색 → GitHub 기본 실패 메일도 병행(보너스, 보장 아님 `[검증 필요·무해]`)

## 2. 과거 부류 (errors.md)
- 2026-06-10 "push|tail 파이프가 exit code 삼킴" → 알림 curl은 `|| true`로 격리하되 **최종 exit 1은 파이프 없이** 직접.
- best-effort 알림이 또 다른 무음 실패가 되지 않게: 알림 실패해도 exit 1은 유지(빨간 run은 남음).

## 3. 추론 가능 vs 실환경 전용
- 머리로 거름: found=no 경로는 기존 guard(`if: found == 'yes'`)로 이후 step이 어차피 skip — 알림 step 추가는 기존 흐름 불변. dry_run 조건, 콤마 파싱.
- 실환경에서만: 실제 생성 실패 재현은 인위적으로 안 만듦(라이브 대기). 대신 **알림 curl 명령 자체는 기존 검증된 Bot API 패턴과 동일**이라 스모크 생략 — 첫 실패 발생 시 도착으로 최종 확인.

## 4. 검증
- 자동: YAML 파싱(ruby) + walk-through(아래) + 기존 dry_run 경로에 영향 없음 확인.
- walk-through: schedule KR run에서 Claude가 파일 미생성 → `found=no` → 알림 step 조건 `already==no && found==no && dry_run!=true` 전부 참 → curl sendMessage × chat 2 → exit 1 → run 빨강, Send/Commit은 기존 guard로 skip(변화 없음).
- 사용자: 실패가 실제 발생한 날 텔레그램에 "⚠️ … 리포트 생성 실패" 도착(그 전엔 확인 불가·정상).
