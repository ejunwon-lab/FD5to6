## 1. 외부 동작 가정 + 근거

- **daisy-chain은 주말에도 24/7 alive** → 일요일 13:0x 창을 감지해 dispatch 가능 → `telegram-push.yml:19` 주석 "24/7 체인: 장외·주말 poke" + self-dispatch 120분 루프(`:110-125`)가 요일 무관 연속. **단, 주말엔 cron 재시드 백업이 없음**(`:28` cron은 `* * 1-5` 평일만) → 금요일 마지막 재시드(~16:40 KST) 이후 일요일 13:00까지 ~22링크 순수 self-dispatch. 러너 사망 시 월요일 09:00 cron까지 재시드 0. **[실환경 검증 필요 — 다음 일요일 체인 생존율]**
- **cron `2 4 * * 0`(UTC) = 일요일 13:02 KST** → cron dow 0=일요일, 04:00 UTC + 9h = 13:00 KST 같은 날(주말 경계 안 넘음). 사실(시간대 산술).
- **`gh workflow run ... -f type=weekly -f auto=true`가 weekly-report job만 발화** → job `if`를 `inputs.type == 'weekly'`로 게이트. 기존 us/kr job과 동일 구조(`:48-50`). 사실(YAML 의미).
- **send_telegram.py·이메일 step은 리포트 파일경로만 받는 범용** → `send_telegram.py`는 `sys.argv[1]` 파일경로 1개만 의존(종류 무관). 이메일 step도 `--rawfile t <path>`로 본문 직렬화. 사실(코드 확인 `send_telegram.py:66`, `market-report.yml:140`).
- **Claude CLI가 5일치 리포트·파이프라인·GAS지표를 종합 가능** → 에이전트는 WebFetch/Bash/Read/Write 보유(`:97`,`:209`). `docs/reports/KR-*.md`·`US-*.md`·`_kr_pipeline.md` Read, `$GAS_WEB_APP_URL`로 `action=portfolioMetrics` curl(`kr-prompt.md:75-78`), WebSearch로 차주 이벤트. 사실(기존 kr-prompt가 동일 도구로 동작 중).
- **멱등: `docs/reports/WEEK-{일요일}.md` 존재 시 schedule·auto dispatch는 skip** → 평일 가드(`:60-74`) 동일 적용. 체인 dispatch + cron 백업이 거의 동시 발화해도 둘 중 하나만 발송. 사실(패턴 복제).
- **휴장/데이터 공백은 일요일엔 상시** → 일요일은 항상 비거래일. 주간 리포트는 "지난 주 마감 데이터 종합"이므로 당일 시장 데이터 불필요(금요일까지 5거래일 데이터로 충분). 설계상 무관.

## 2. 과거 부류 (errors.md) — 매칭 & 회피책

- **errors.md:23-32 (GitHub cron best-effort, 고빈도 cron 통째 드롭 → daisy-chain)** → 교훈 "best-effort 의존 횟수를 줄여라". **회피**: 주경로를 cron이 아닌 **체인 dispatch(이벤트 구동, 6/8~19 드롭 0)**로. cron은 백업으로만. 단 주말 체인-liveness 갭(아래 3)은 이 회피의 약점 → 일요일 아침 reseed 틱으로 보강 검토.
- **errors.md:47-52 ("2회 안전망 race" — 지연된 2 cron이 9분차 동시 실행, push 충돌로 1개 rejected)** → **회피**: ① `concurrency: group: market-report-weekly`(1 running + 1 pending 수렴) ② 파일존재 가드가 pending run에서 재평가 → 단일 발송. 평일 us/kr이 이 조합으로 이미 race 무사고. weekly도 동일.
- **errors.md:12 (멱등 키 체크가 다른 게이트보다 먼저 와야)** → 가드 step(`guard`)을 모든 생성·발송 step보다 앞(`:60`)에 둔 기존 순서 유지.
- **octal 파싱 버그(`10#` — telegram-push.yml:85 주석)** → 일요일 창 비교도 `hm=$((10#$(date +%H%M)))` 그대로 사용(1302엔 leading-zero 없지만 일관성·안전).
- **errors.md:317-334 (주말·비거래일에 시장 데이터 0/오염)** → 회피: 주간 리포트는 당일(일요일) 시장 데이터를 **쓰지 않음**. 금요일까지 5거래일 일일 리포트를 소스로 사용 → 비거래일 오염 무관.

→ 과거 4부류 전부 기존 패턴 복제로 회피. **신규 위험은 "주말 체인 liveness" 하나뿐** (평일 패턴엔 없던 갭).

## 3. 추론 가능 vs 실환경 전용

**머리로 거름 (지금 검증):**
- YAML 유효성·job if 게이트·cron→KST 매핑(13:02) → syntax check + 산술.
- 일요일 dow 판정(`date +%u`=7) + 창(1302~1359) 비교 walk-through.
- 멱등 가드 walk-through: 체인 dispatch와 cron이 동시 발화 → 먼저 끝난 쪽이 WEEK 파일 commit → 나머지는 가드 skip 또는 concurrency pending에서 파일 보고 skip → 단일 발송.
- 프롬프트가 소스(5일 리포트·파이프라인·GAS)를 정확히 가리키는지 → 파일 존재·엔드포인트 확인.

**실환경에서만 (스모크/실측):**
- **다음 일요일 13:00에 체인이 실제 살아있나** → 주말 ~44h 무재시드 구간 생존율. 로컬·평일 테스트로 "됨" 판정 **금지**. → 첫 일요일 실측.
- **cron `2 4 * * 0`이 일요일 13:0x에 실제 발화하나** (저빈도 cron은 발화하나 1~3h 지연 전례 errors.md:31) → 첫 일요일 실측.
- **이메일 실발송 HTML 도착** (GAS emailReport 채널) → 첫 일요일 실측(PB 리포트 검증과 동일).
- **리포트 내용 품질** (리캡이 유용한 종합인지) → dry_run 본문 검토 + 사용자 피드백.

## 4. 검증 방법

**자동 (코딩 직후):**
- `ruby -ryaml -e 'YAML.load_file(...)'` → market-report.yml·telegram-push.yml 파싱 OK.
- `bash -n` 또는 추출 후 검사 → 수정한 run 블록 문법 OK.
- 일요일 창 판정 walk-through: `dow=7, hm=1302` → `[ 7 -eq 7 ] && [ 1302 -ge 1302 ] && [ 1302 -le 1359 ]` → `maybe_dispatch_report weekly WEEK` 호출 ✓. `hm=1300`(13:00 정각, 창밖) → skip, 다음 poke(13:05)에 진입 ✓.
- 멱등 walk-through: WEEK 파일 존재 + auto dispatch → `GUARD_ON=yes`,`-f`존재 → `already=yes` → 생성·발송·commit 전부 skip ✓.

- **dry_run 게이트 (라이브 전 필수, PB 리포트와 동일 방식):**
  `gh workflow run "Market Report (US/KR)" -f type=weekly -f dry_run=true` → run 로그에서
  ① WEEK 파일 생성(`found=yes`) ② 본문에 지난 주 5거래일 리캡 + 차주 계획 + 포트 지표 ③ 발송·commit은 skip(dry_run) 확인.
