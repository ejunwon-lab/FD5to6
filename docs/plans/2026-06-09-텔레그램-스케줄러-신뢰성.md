## 1. 외부 동작 가정 + 근거

- **GitHub Actions `schedule`은 best-effort — 고부하 시 지연·드롭, 정시 보장 없음.**
  → 근거: 위 실측(2/21) + GitHub 공식 문서("scheduled workflows may be delayed during periods of high load") + errors.md 2026-06-09. **사실 확정.**
- **격자 변경(:00 → :05/:25/:45)으론 안 고쳐진다.**
  → 근거: 그 격자가 이미 적용된 상태에서 2/21 → **이미 실패한 대책.** 추가 격자 튜닝은 무의미.
- **`tgPushPnL`은 매 호출마다 무조건 발송한다 (시간 dedup 없음).**
  → 근거: [Telegram.js:139-174](../../apps-script-v2/Telegram.js#L139) 직접 읽음. 거래일·09~16시·락 통과하면 `tgSendMessage` 실행. 락은 `tryLock(1000)` 동시성만 차단, 순차(30분 간격) 중복은 못 막음. update_id dedup(L203-210)은 *webhook 전용*이라 무관.
  → **귀결: 두 시계가 같은 창에서 둘 다 호출하면 메시지 2번 간다.** redundancy 설계는 **dedup 추가가 전제**.
- **과발화는 무해, 과소발화만 실패.**
  → 근거: 거래일·시간·락 자체 체크(코드). → over-provision(시계 더 자주)이 안전한 방향.
- **Cloudflare Worker Cron이 GitHub보다 sub-hourly 신뢰성이 높다.** → `[검증 필요·실환경 전용]`
  → Cloudflare 문서도 cron은 best-effort라 명시. "더 낫다"는 **추측** — 실제 적중률은 프로덕션 측정 필수.
- **기존 Worker(Telegram proxy) 재사용으로 cron 추가 가능.** → `[검증 필요]`
  → cron trigger 추가가 wrangler 배포·version promote 흐름에 영향. memory [[feedback_cloudflare_worker_deploy]]: secret/version 변경은 Deployments 탭 수동 promote 필요.

## 2. 과거 부류 (errors.md) → 회피책

- **errors.md 2026-06-06**: GAS 트리거 best-effort → GitHub Actions "신뢰 시계"로 교체했는데 **GitHub도 같은 병**. → **회피책: 새 시계(Cloudflare)를 "문서가 신뢰성 좋다더라"로 채택 절대 금지. 1+ 거래일 실측 적중률 숫자 게이트 통과 후에만 프로덕션 전환.**
- **errors.md 2026-06-09 (KR 리포트)**: "Naver 차단"이 LLM 추측이었고 실패 모드 미확정. → 회피책: "Cloudflare 신뢰" 동일하게 추측 금지, 실측으로만.
- **"미국 IP 프로브 200" 무효 검증**: 프로덕션과 다른 환경 테스트는 무효. → 회피책: Cloudflare 측정은 **실제 Worker 계정·cron에서**. 로컬·다른 환경 대체 금지.
- **errors.md 2026-06-09 (KR race)**: 안전망 2회가 스케줄러 지연으로 시간 겹쳐 push race. 멱등을 *시각 간격*에 의존한 게 원인. → **직접 연결: redundancy(두 시계) 설계 시 dedup을 시각 간격이 아니라 명시적 상태(PropertiesService lastPushMin)로.** 1번 가정과 일치.
- **🔴 errors.md 2026-05-31 (66-72) — Cloudflare PoP IP → GAS 403 (확정·치명적)**: Cloudflare Worker outbound는 *실행 PoP의 IP*로 나가고, **미국 PoP IP → GAS 웹앱 호출이 Google anti-abuse에 `403 Host not in allowlist`로 차단**됨(한국 PoP만 통과). → **Cloudflare cron이 GAS `pushPnL`을 부르면 caller 없는 스케줄 실행이라 PoP를 못 고르고, 미국 PoP면 그날 0건 전달.** 사용자 지적("Cloudflare가 못해서 GitHub 썼냐")의 실체 = Cloudflare는 *proxy*로 실패했고 그 원인이 **PoP IP→GAS 연결 차단**. GitHub 러너는 같은 GAS를 *실제로 도달*(현재 success run 존재)하므로 연결은 입증됨 — Cloudflare는 아님.
  → **회피책: 후보 시계는 "타이밍 적중률"보다 먼저 "GAS 연결되나"를 게이트해야 한다.** GitHub=연결✅/타이밍❌, Cloudflare=연결⚠️(403 전례)/타이밍❓. best-effort(견딜 만함)를 403(치명적)과 맞바꾸지 말 것.

## 3. 추론 가능 vs 실환경 전용

**머리로 거름 (여기서 확정):**
- GitHub 21/21 기대는 구조적으로 불가 — best-effort 확정.
- 격자 튜닝은 무효 (실측이 증명).
- tgPushPnL 다중 호출 = 다중 발송 (코드 확정) → redundancy엔 dedup 필수.
- 과발화 무해 / 과소발화만 실패 → over-provision 안전.
- **목표치(21회 vs 핵심 3~4회)는 상위 결정** — 적으면 핵심 슬롯 둘레에 틱 클러스터 + 멱등으로 best-effort로도 충분, 많으면 진짜 신뢰 시계 필요. → **사용자 결정 사항(아래).**

**실환경에서만 (스모크/실측으로):**
- Cloudflare Worker Cron 실제 :05/:25/:45 적중률 → 1+ 거래일 트라이얼.
- GitHub 드롭이 *윈도우 통째* vs *틱 랜덤* (n=2일론 부족) → redundancy 효과(union 적중률) 예측하려면 더 측정.
- GAS 웹앱을 더 자주 호출 시 latency/quota 영향 → 측정.

## 게이트 판정: **통과** (코딩 진입 가능) — 단, 두 상위 결정이 코딩 방향을 가른다

구현 착수 전 확정 필요:
1. **목표치** — 21회(촘촘) vs 핵심 3~4회(개장·중간·마감). **사용자 결정.** 이게 "신뢰 시계가 필요한가, 클러스터+멱등으로 충분한가"를 가름.
2. **단일 교체 vs redundancy** — Cloudflare로 GitHub 교체(단순, GitHub 부분 적중 포기) vs 두 시계 병행(union 적중률↑, **dedup 추가 필수**).

이 둘 미확정 상태에서 코드를 짜면 방향이 틀릴 수 있으므로, 설계 노트는 통과지만 **(1)(2) 결정 후 구현**.

### 사용자 결정 (확정)
- **목표치 = 21회 촘촘히.** → dedup-only로는 적중률이 안 오르므로 GitHub보다 정시를 잘 지키는 시계가 필요.

### 권고 (분석 기반·2026-06-09 사용자 지적 반영해 개정)
**개정 이유**: 사용자가 "Cloudflare가 못해서 GitHub 쓴 것 아니냐"고 지적 → errors.md 검증 결과, Cloudflare는 *시계*로 쓴 적 없고 *proxy*로 실패했으며 그 원인이 **PoP IP→GAS 403 연결 차단**(errors.md 2026-05-31). 즉 Cloudflare cron은 GAS 연결 자체가 막힐 전례가 있어, GitHub의 *타이밍* 문제보다 더 나쁜 *0건 전달* 위험을 안음. GitHub 러너는 GAS 연결이 입증됨.

→ **개정 권고: "교체 후보를 찾기" 전에, 연결이 입증된 GitHub의 타이밍을 먼저 강화 시도.**
1. **1순위 (연결 위험 0)**: GitHub cron 틱 밀도↑ (예 `*/5` 윈도우 내) + tgPushPnL에 N분 dedup → 드롭 많아도 더 많이 착지하는지 **1 거래일 실측**. GitHub 드롭이 *윈도우 통째*면 효과 없고(밀도 무의미), *틱 랜덤*이면 효과 큼 — 이게 `[실측 필요]`의 핵심.
2. **2순위 (밀도↑가 부족할 때만)**: 외부 cron 후보. 단 **연결 게이트 먼저** — 후보가 GAS `pushPnL`에 실제 200 받는지(특히 Cloudflare는 PoP IP 403 전례) 1회 확인 *후에* 타이밍 측정. 연결 실패면 후보 탈락.
3. Cloudflare cron은 PoP→GAS 403 전례 때문에 **연결 게이트를 통과해야만** 후보 자격. 통과 못 하면 cron-job.org·Google Cloud Scheduler 등 연결 입증되는 외부 cron으로.

→ 핵심 원칙: **타이밍 적중률을 재기 전에 "이 시계가 GAS에 닿나"를 먼저 게이트한다.** (errors.md 2026-06-06 "측정 없이 새 시계 신뢰" + 2026-05-31 "PoP 403" 두 함정 동시 회피)

## 채택 설계 — F (단일 신뢰 시작 + 자체 루프) 구현 (2026-06-09 확정)

### GAS — `tgPushPnL`에 18분 dedup (Telegram.js)
- 락 안에서 `tg_lastPushEpoch`(ScriptProperties) 읽어 직전 발송 <18분이면 skip. **발송 전 슬롯 선점**(updateNewPriceHistory 전) → KIS 과호출·중복 발송 차단.
- `tgRefreshAndPush`(수동 ㄱㄱ)도 발송 후 `tg_lastPushEpoch` 갱신 → 수동+자동 중복 방지. (dedup은 `tgPushPnL`에만, 수동은 항상 즉시 발송)
- 카덴스 = **GAS가 단일 권위**. poke 5분 + dedup 18분 → 첫 poke≥18분이 +20분 grid에 착지 → ~20분 카덴스(09:00~16:00 ≈ 22회). 루프 개수·오프셋 무관.

### GitHub — `telegram-push.yml` 재작성 (시작만 담당)
- 잡 1개 = `while UTC<END; do curl pushPnL; sleep 300; done`, 230분 지속(6h 상한 회피, timeout 245).
- cron 4개(UTC): shift A `0 0`·`3 0`, shift B `30 3`·`33 3` (월~금). 커버 00:00~07:20 UTC = 09:00~16:20 KST(GAS가 16:00에 컷).
- redundancy 3종: 중복 시작 틱(시작 드롭)·2 shift(6h 상한)·겹치는 잡(러너 사망). GAS dedup이 겹침 무해화 → **concurrency 제어 없음**(분 무료).

### 🔴 배포 의존성 (블로킹)
`push_safe.py`는 코드만 PUT → 웹앱 `/exec`은 **새 배포 버전** 전엔 옛 `tgPushPnL`(dedup 없음) 서비스 → 5분 poke가 **5분마다 스팸**. **재배포(GAS 에디터 배포 관리 → 신규 버전) 전엔 cron 활성 위험.** 6/10 09:00 KST cron 전에 재배포 필수.
