# 2026-07-23 — Analysis 리스크 지표 자산 기반 교정 설계 노트 (/design-check)

데스크 로드맵 ⑥. Sharpe·Volatility·Win Days·Best Day·MDD가 **누적수익 곡선의 일별 %변화** 기반 → 분모가 자산이 아니라 누적수익이라 과대 (수익 1,000만에서 +50만 = "+5%"로 계산, 자산 기준 실제는 훨씬 작음).

## 수정 설계

- **GAS**: `newMobileGetProfitHistory` 응답 entries에 `totalAsset` 필드 추가 — *추이 기록* (B) 일별 추이 **Q열(합계 총자산 = 운용+대기)**을 날짜 키로 매칭해 부착. 기존 필드 불변(additive) → web·iOS 후방 호환. 윈도는 기존 슬라이스 그대로.
- **desk**: `mapEquity`가 `asset` 보존 → `computeStats`의 일별 수익률·MDD·Sharpe·Vol·Win/Best를 **자산 시리즈** 기반으로 교체. 자산 데이터 부족(샘플 폴백 등) 시 기존 수익곡선 방식 유지(폴백).

## 1. 외부 동작 가정 + 근거

- Q열 = `sumTotal`(opTotalCur+pendTotalCur), 일별 upsert — Trend.js (C)와 같은 함수에서 기록되므로 날짜 집합이 수익 추이(U열)와 사실상 동일. 불일치 날짜는 매칭 실패 시 `totalAsset: null` → 클라이언트가 해당 구간 skip.
- 주말 행: 수익 추이 쪽 `_isTradingDateStr` 필터가 이미 적용된 날짜에만 매칭하므로 자동 배제.
- MDD 정의가 GAS `getPortfolioMetrics`(공식 리포트 경로)의 "Q열 peak-to-trough"와 일치하게 됨 — 두 시스템 간 정의 통일.

## 2. 과거 부류 (errors.md)

- **TWR 입금 왜곡 (2026-07-16)** — 자산 diff는 입출금을 수익으로 오인. read-time flow 보정은 doPost `getPortfolioMetrics` 전용이라 이 경로엔 없음. **한계로 명시**: 데스크 리스크 지표는 raw 자산 수익률 기반(입출금일 왜곡 잔존 가능) — 그래도 "수익곡선 %변화"보다 자릿수 단위로 정확. 완전 TWR은 후속(보정 로직 공용화) 과제.
- % 셀 비대칭(2026-07-16) — Q열은 금액 numeric, % 아님 → 해당 없음.

## 3. 추론 가능 vs 실환경 전용

- 머리로 거름: 필드 부착·매칭·클라 계산 교체 — walk-through로 검증.
- 실환경에서만: 실데이터에서 Sharpe·Vol 값이 "그럴듯한 범위"(일 σ ≲ 2%대)로 내려오는지 → 사용자 화면 확인 이관.
