# 설계 노트 — 매도 종목 What-if 추적 + 기간별 '번 돈' 타일

> 2026-07-15. 위험 변경 후보(GAS 서버코드·스케줄 편승·신규 read API) → `/design-check`.
> 요청: (1) 판 종목을 별도 시트에 모아 매일 당일가격 → "안 팔았다면 손익" vs "실제 실현손익" vs "판 것 대비 차이". 과거 전체 매도 소급. 시트+웹+데스크. (2) 기간별 번 돈(1주·1개월·YTD, 실현+평가) 타일.

## 핵심 재구성 (조사로 위험 소멸)

당초 "KIS 신규 호출 + 별도 일별 append 시트"로 생각했으나, 조사 결과:

- **`updateNewPriceHistory`는 *거래_원장* 전체 코드에서 종목을 뽑는다**([NewSystem.js:830](../../apps-script-v2/NewSystem.js)). 판 종목도 원장 매수/매도 행이 남아 **코드가 codeSet에 계속 포함** → *현재가_이력*이 **판 종목 daily 가격을 이미 매일 기록 중**.
- 백업 실측(2026-07-14): *현재가_이력* 헤더에 어제 매도 4건(396500·0047A0·495230·471990) + 옛 매도(196170·228790·390390·445290) + **해외 MU**까지 전부 열로 존재. 예금(신한정기예금)만 없음(코드 없음·KIS_SKIP).
- *현재가_이력* 커버리지: **2026-01-11 ~ 현재**(129행).

→ **결론: Feature B는 신규 외부 호출·신규 append 저장 없이 `실현손익` + `현재가_이력` 두 시트의 파생으로 구현**. 일별 이력의 SSOT는 이미 *현재가_이력*(전역 스키마 원칙의 "이력은 append"를 이미 충족) — 중복 저장 안 함. **매도추적 시트는 매도 이벤트당 1행 스냅샷(state, 매 갱신 재계산)**.

## 소급 커버리지 (사용자 결정 "과거 전체 소급"에 대한 실측 답)

| 산출물 | 소급 범위 | 근거 |
|---|---|---|
| **오늘 스냅샷** (안팔았다면·판것대비 차이) — 핵심 가치 | **48건 전부**(예금 제외 47) | 오늘 가격은 모든 판 종목에 존재 |
| **매도일부터 일별 후회 곡선** | max(매도일, 2026-01-11) 이후 | *현재가_이력* 시작 2026-01-11 |

- 2026-01-11 **이전 매도**(48건 중 다수)의 곡선은 매도일이 아니라 2026-01-11부터 시작. 스냅샷 값은 정확.
- 이 gap을 매도일까지 채우려면 **KIS 국내 일봉 소급**(FHKST03010100, [KIS_API.js:446](../../apps-script-v2/KIS_API.js)) 필요 = 유일한 고위험 요소 → **v2로 분리, v1은 파생만**. 권고: v1 먼저(핵심 가치 100% 달성), 곡선 소급은 효용 확인 후.

## 1. 외부 동작 가정 + 근거

- *현재가_이력*이 판 종목 가격을 계속 채운다 → **근거**: [NewSystem.js:830](../../apps-script-v2/NewSystem.js) 원장 기반 codeSet + 백업 헤더 실측. KIS 현재가는 보유 무관 종목코드 조회라 매도 후에도 응답.
- 매도추적 계산이 `updateAllNew` 체인에 편승 → **근거**: [Main.js:118](../../apps-script-v2/Main.js) `updateFxRates→updateNewPriceHistory→updatePositionFromLedger→logToTrendSheet→buildDashboard`. `computeStockMetrics`처럼 `updatePositionFromLedger` 끝 또는 체인 말미에 `buildSoldTracker()` 1개 추가. **신규 트리거 없음**.
- 웹/데스크 신규 read = 기존 `newMobileGet*` + `scripts.run` 패턴 재사용 → **근거**: [gasApi.ts](../../web/src/api/gasApi.ts) `callGAS('newMobileGetPortfolio')`. 신규 `newMobileGetSoldTracker` 동일 패턴.
- 기간별 번 돈 = 추이 AD열 diff → **근거**: `newMobileGetProfitHistory` 이미 `entries[{date,totalProfit}]` 반환([MobileAPI.js:746](../../apps-script-v2/MobileAPI.js)). AD=확정누적+평가라 기간 diff에 실현 자동 포함. **순수 클라 함수**(GAS 무변경).
- 실현손익 스키마 = `[0]매도일 [1]코드 [2]종목명 [3]분류 [6]매도수량 [8]매도금액 [10]매입원가 [11]수수료 [12]실현손익` → **근거**: code-map.md:70. 예금 행은 [1]코드 빈칸 → 필터.

## 2. 과거 부류 (errors.md) + 회피

- **비거래일 행 누적**(2026-05-17 계열·errors.md 201·354·367): 시계열에 주말/공휴일 행 끼면 change 오염. → 회피: 매도추적은 *현재가_이력*의 **마지막 거래일 행**(priceAsOfDate 패턴)만 읽음. 신규 웹 endpoint도 `_isTradingDateStr` 필터(newMobileGetProfitHistory와 동일).
- **KIS carry-forward stale**(errors.md 9·10): KIS 전면 실패 시 전일가 복제 → change=0. → 매도추적은 절대금액 스냅샷이라 carry면 "차이"가 전일과 동일(치명적 아님). `kis_carried_status` 이미 존재 → 필요 시 stale 배지. v1은 무표기(허용).
- **updateAllNew 무거움/타임아웃**(errors.md 105·257): 22~83초, 무거우면 트리거 지연·doPost retry. → 회피: 매도추적은 **KIS 호출 0**(시트 읽기+산술만) → 체인에 붙여도 부하 미미. **소급 KIS 일봉(v2)은 doPost/트리거에 절대 안 붙임** — 별도 1회 수동 함수로.
- **부분 갱신이 표시 출처 시트 안 건드림**(errors.md 70·72): 표시값의 실제 출처 역추적 필요. → 매도추적 출처 = 매도추적 시트(state), `buildSoldTracker`는 `updateAllNew`에만 → `tgPushPnL`도 `updateAllNew` 호출하므로(errors.md 73 해결로 통일됨) 항상 fresh.
- **원장 직접 타이핑 시 미재계산**(errors.md 379): 매도추적도 재계산 트리거 필요 → `updateAllNew`/⚡ 경유만 신뢰(수기 원장 편집 후엔 재계산 필요, 기존과 동일).

## 3. 추론 가능 vs 실환경 전용

- **머리로 거름(여기서 확정)**:
  - 파생 산식: 안팔았다면손익 = 현재가×매도수량 − 매입원가; 판것대비차이 = 안팔았다면손익 − 실현손익 = (현재가−매도가)×수량 + 수수료. 검산: 어제 396500 매도가[금액]·현재[금액]·504주 → 차이=([금액]−[금액])×504+795 ≈ +[금액](안 팔았으면 소폭 더). ✓ 부호 의미: +면 "안 팔걸", −면 "잘 팔았다".
  - 예금/펀드 제외(코드 빈칸 필터), 해외 MU 포함(현재가_이력에 존재).
  - 매도 이벤트 키 = (매도일·코드·증권사·계좌) — 같은 종목 여러 매도 분리(다계좌·재매도 함정, errors.md 306 계열).
  - 비거래일 가드·기간 diff 실현 포함 로직.
- **실환경에서만(배포 후 실측)**:
  - `buildSoldTracker`가 실제 *현재가_이력* 오늘 열 인덱스를 정확히 매핑해 47건 값 채우는지(코드 정규화 `_normCode` 0-prefix·A-prefix 함정).
  - `updateAllNew` 총 소요가 매도추적 추가로 유의미하게 안 늘어나는지(타임아웃 여유).
  - 웹/데스크 신규 카드가 실데이터로 렌더되는지(스냅샷 부호·정렬).

## 게이트 판정

- **통과 (v1 = 파생 전용)**. 고위험 요소(KIS 신규 호출·신규 스케줄·타임아웃)가 조사로 **소멸** — 남은 건 read 함수 1개 + state 시트 1개 + 클라 컴포넌트(중저위험).
- **1건 사용자 결정 남음**: 2026-01-11 이전 매도의 **일별 곡선 gap**을 (a) 그대로 수용(v1, 권고) vs (b) KIS 일봉 소급(v2, 별도 수동 함수). 스냅샷 핵심 가치는 (a)로 100% 달성.

## 구현 순서 (승인 후)

1. **GAS** `buildSoldTracker()` — *실현손익*×*현재가_이력* 파생 → *매도추적* 시트(state). `updateAllNew` 말미 호출. `newMobileGetSoldTracker()` read 함수.
