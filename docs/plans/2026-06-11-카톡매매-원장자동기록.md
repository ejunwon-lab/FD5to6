## 핵심: "원장에 한 줄 append"가 전부. 진짜 문제는 로컬에서 시트에 쓰는 경로 (실측 완료)
| 경로 | 결과 | 비고 |
|---|---|---|
| 직접 Sheets API write | ❌ | clasp 토큰에 `spreadsheets` 스코프 없음(`drive.file`뿐) |
| `scripts.run`(함수 직접 실행) | ❌ 403 | 토큰 스코프 부족(spreadsheets/scriptapp 없음). 재로그인 과함 |
| **doPost 웹앱 POST** | ✅ | 검증된 경로(errors.md:107). 코드는 push_safe로 배포, URL은 API로 획득 |
| push_safe 배포 | ✅ | 토큰에 `script.projects` 있음 — `_buildDiag` 등 푸시 가능 |
| deployments API로 URL 획득 | ✅ 실측 | HEAD 웹앱 `.../AKfycbxFAT...e/exec`(항상 최신). 사용자 입력 불필요 |

→ **사용자에게 받을 단 하나: `TG_WEBHOOK_SECRET`**(ANYONE_ANONYMOUS 엔드포인트라 ledger-write 보호 필수). gitignore된 로컬 파일에 1회 저장.

## 아키텍처 (confirm-then-write) — 파싱=Claude, 권위적 쓰기=GAS
1. 사용자가 카톡 붙여넣음.
2. **Claude 파싱**(messy NL + 증권사별 포맷): `{날짜,구분,코드,종목명,분류?,증권사,계좌,수량,단가,금액,수수료?,주문번호}`. 정규화: `미래에셋증권`→`미래에셋투자증권`, `A257720`→`257720`, 날짜 미지정→오늘(KST), 금액 미지정→수량×단가. 마스킹 계좌→매핑 memory로 실계좌. 수수료=계좌 규칙(모르면 질문).
3. **Claude가 파싱 행 표시 → 사용자 확인.**
4. Claude가 구조화 JSON + secret을 HEAD `/exec`에 POST(`action=addTrade`).
5. **GAS `_handleAddTradePost`**(권위적): secret 검증 → 멱등 체크 → 분류 미지정 시 보유현황 룩업 → 검증 → 원장 1행 append → `updatePositionFromLedger` → `{ok,row,beforeQty,afterQty}`.
6. Claude가 전후 수량으로 결과 보고.

## 1. 외부 동작 가정 (전부 실측/문서 근거)
- doPost `action` 분기에 `addTrade` 추가 가능 → **근거**: Main.js:76 기존 분기(pushPnL 등).
- 로컬→HEAD `/exec` POST 200 가능 → **근거**: errors.md:107 로컬 curl 200 전례 + PoP403은 US IP 한정(115).
- curl `-X POST` 금지, `--data`+`-L` → **근거**: errors.md:42-43.
- secret 검증 = `TG_WEBHOOK_SECRET` 재사용, 불일치 시 forbidden → **근거**: errors.md:44, code-map.md:177.
- `updatePositionFromLedger`가 전체 재계산 → **근거**: code-map.md:126.
- 웹앱 URL을 deployments API로 획득 → **실측 완료**(이 세션). HEAD는 push_safe PUT 직후 최신 반영.

## 2. 과거 부류 (errors.md) + 회피
- **update_id 멱등 (203-215)**: doPost retry 가능 → 멱등 키 필수. → 회피: append 전 원장에서 `(날짜,증권사,계좌,코드,구분,수량,단가)+메모 주문번호` 동일 행 있으면 skip. 부분체결(같은 주문번호·다른 수량)은 키 달라 별개 보존.
- **다계좌 granularity (306-311)**: 보유현황 행은 `(코드,증권사,계좌)`. 같은 종목 2계좌 정상. → 회피: 분류 룩업·전후수량을 **코드 단독 아닌 코드+증권사+계좌** 키로.
- **무거운 doPost 타임아웃 (205)**: 전체 재계산 수초 → 응답 지연/ retry. → 회피: `LockService`로 tgPushPnL·트리거와 충돌 차단 + confirm-then-write(1회 POST) + 멱등이 retry 중복 흡수.

## 3. 추론 가능 vs 실환경 전용
- **머리로 거름**: 파싱·정규화(검산 `430×34,900=15,007,000`✓), 멱등 키, 분류 룩업 granularity, secret/302 규칙, LockService.
- **실환경에서만 (배포 후 스모크 필수)**: ① 실 secret POST 200+ok ② 원장 실제 1행 추가+보유현황 수량 감소 ③ 매도 분류 룩업이 보유현황서 정답 찾음 ④ 멱등: 동일 fill 2회→1행 ⑤ 마스킹→실계좌 매핑 정확.

## 4. 검증 방법
- **자동(코딩 후)**: push_safe `node --check`; 더미 secret POST→`forbidden`; 파싱 walk-through(실리콘투 샘플→12필드); 멱등 walk-through(주문번호 2회→2번째 skip).
- **🔴 스모크(배포 후 1회, 실 secret)**: 샘플 1건 POST → `ok` + afterQty=beforeQty−430 → 재POST → `already`(중복 0).
- **🔴 사용자 확인**: *보유현황* 시트 **실리콘투(257720)·미래/종합_랩 보유수량 −430**, *실현손익* 새 행(매도 430·34,900), *거래_원장* 새 행 + 메모 `주문번호:6396`.

→ **게이트: 보류 → 사용자 입력 2건 받으면 통과.** ① `TG_WEBHOOK_SECRET`(GAS `tgShowSecret`) ② `종합_랩` 수수료 규칙. 파싱·GAS 로직·접근경로는 설계·실측 완료.
