# /design-check — 코딩 전 설계 게이트

위험 변경을 **코딩하기 전에** 호출한다. 목적: 재작업 루프(디버깅→재코딩→재배포→재검증)를 코딩 전 사고 1회로 대체.

## 언제 쓰나 (위험 변경 트리거)

아래 중 하나라도 건드리면 필수. 아니면 건너뛴다(순수 함수·UI·문서는 불필요).

- 배포·실행되는 **서버 사이드 코드** (요청 핸들러·스케줄 작업·트리거 진입점)
- **CI/CD·스케줄 파이프라인** (cron·워크플로·배포 스크립트)
- **권한·인증·접근 설정** (스코프·공개 범위·시크릿·토큰)
- **외부 API/서비스 호출** 추가·변경

판단 모호하면: **"외부 의존이 결과를 바꾸나?"** → 그렇다면 위험 변경.

## 절차

### 0. ultrathink 모드로 진입
이 패스는 깊은 사고가 핵심이다. 얕게 통과시키면 의미 없다.

### 1. 과거 에러 먼저 검색 (코드 읽기 전)
프로젝트에 에러 로그(예: `docs/errors.md`)가 있으면 같은 영역/같은 외부 의존이 과거 터졌는지 검색:
```bash
grep -niE "<관련 키워드>" docs/errors.md 2>/dev/null
```
터졌으면 그 **원인과 해결**을 읽고, 이번 설계가 어떻게 그 함정을 피하는지 명시.

### 2. 설계 노트 4항목 작성 (추상 금지, 구체로)

1. **외부 동작 가정 + 근거** — 변경이 기대는 외부 동작을 한 줄씩. 각각 *문서/사실*로 뒷받침하거나 `[검증 필요]` 표시.
2. **과거 부류 매칭** — 1번 검색 결과. 재발 위험과 회피책.
3. **추론으로 아는 것 vs 실환경 검증 필요** — 두 칸으로 갈라 적는다.
   - *머리로 거름*: 리다이렉트 동작·권한 의미·멱등성·발화 보장 여부 등
   - *실환경에서만*: 특정 IP 차단·실제 응답 스키마·스케줄러 실측 적중률 등 → 추론으로 "됨" 판정 금지, 스모크/실측으로만.
   - ⚠️ 자주 하는 실수: 실환경 전용 사실을 다른 환경(로컬·다른 IP) 테스트로 "검증 완료"라 함. 반복 금지.
4. **검증 방법 사전 정의** — 어디서·무엇을·예상값. 자동 검증(syntax check·curl·테스트) + 사용자 확인 분리.

### 3. 산출물 저장
프로젝트에 계획 폴더가 있으면 `docs/plans/YYYY-MM-DD-<주제>.md`에 위 노트를 쓴다(코딩 허가증이자 사후 재참조). 없으면 응답에 인라인.

### 4. 게이트 판정
- 4항목이 구체적으로 채워졌고, 1·2의 과거 함정을 회피했으며, 3에서 "실환경 검증 필요" 항목이 검증 방법(4)에 반영됐는가?
- 통과 → 코딩 시작.
- 가정에 `[검증 필요]`가 남으면 → 코딩 전에 그 검증부터(가능하면), 또는 검증 계획을 명시하고 진행.

## 출력 형식

```
## 설계 노트 — <주제>

### 1. 외부 동작 가정
- <가정> → <근거 / [검증 필요]>

### 2. 과거 부류 (에러 로그)
- <매칭 항목 또는 "없음"> → <회피책>

### 3. 추론 가능 vs 실환경 전용
- 머리로 거름: ...
- 실환경에서만: ... → 스모크/실측으로

### 4. 검증 방법
- 자동: <도구> → 예상
- 사용자: <어디·무엇·어떻게·예상값>

→ 게이트: 통과 / 보류(이유)
```
