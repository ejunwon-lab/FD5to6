옵션 A·B·C를 제시한 뒤 사용자가 "다 별로다/우려된다"고 하면, 같은 프레임 안의 변형(A', A'')을 더 찾지 말고 **프레임 자체를 의심**해야 한다.

**Why:** 2026-05-23 애플워치 손익 표시 건. "워치 → GAS 직접 호출"이라는 초기 전제 안에서만 옵션을 굴려서 (단축어 시크릿 / 네이티브 앱 / 포기) 세 갈래만 보였음. 사용자가 보안 우려를 짚을 때도 합리화부터 시도("피해 작다", "$99 내면 된다") — 프레임을 깨야 했는데 안 깼다. 사용자가 "다른 방법 찾아봐"라고 강제로 깨주니 그제야 "워치 → 메신저 봇 → GAS" 우회 패턴이 보였고, 이건 시크릿·만료·비용 셋 다 해결하는 깔끔한 답이었다. 이걸 사용자가 짚어주기 전에 내가 먼저 했어야 한다.

**How to apply:**
- 옵션 제시 후 사용자가 모두 거부하면 **변형 찾기 금지**. 먼저 "내가 어떤 전제에 갇혀 있나?"를 묻는다
- "X가 Y를 직접 한다"는 전제는 거의 항상 우회 가능 — 중간 서버, 메신저 봇, 푸시, 큐, 캐시 등
- 사용자 우려를 합리화로 받지 말고 "맞다, 이건 약점이다"로 먼저 인정한 뒤 프레임 재구성
- 옵션 카탈로그 만들기 전에 한 단계 위 질문: "이 작업을 *반드시* X 위치에서 해야 하나? 다른 곳에서 하면?"
