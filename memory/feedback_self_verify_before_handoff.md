작업을 사용자에게 넘기기 전에 **"이거 내가 직접 할 수 있나?"를 먼저 따진다.** 손에 쥔 도구(`gh`·`curl`·API 등)로 가능한 일을 습관적으로 "사용자가 하는 일"이라며 떠넘기지 않는다. 내가 만든 코드·명령·워크플로는 **사용자에게 테스트시키기 전에 내가 직접 실행해 동작을 확인**하고, 버그를 사용자가 발견하게 두지 않는다.

**Why:**
- telegram-push.yml에 `-X POST` 버그를 넣고 "Run workflow 눌러 확인하세요"라고 시켰는데 그 테스트가 *내 버그*로 실패. `gh`·`curl`로 직접 검증할 수단이 있었는데 안 씀.
- 사용자가 두 번 지적: ① "버그 만들어서 나한테 일 시킨다" ② "왜 니가 직접 할 수 있는 걸 나한테 실행·보고하라고 하냐". 원인 = 도구로 *어디까지 내가 할 수 있나*를 점검 안 하고 익숙한 "사용자에게 넘기기" 기본값으로 빠짐 (게으른 기본값).

**How to apply:**
- 넘기려는 단계마다 먼저 자문: `gh`(워크플로 실행·로그), `curl`/API(외부 호출·데이터 검증), 스크립트로 *내가* 할 수 있나? 가능하면 내가 한다.
- 워크플로: `gh workflow run <file> --ref main` → `gh run view <id> --log`로 직접 실행·확인 후 보고.
- 외부 호출: `curl`로 먼저 때려 응답 형태 확인 (GAS 웹앱은 `-X POST` 금지 — [[feedback_gas_curl_diagnosis]] / errors.md 2026-06-06).
- **정말 사용자만 가능한 것만 넘긴다** — 브라우저 OAuth(`clasp login`), 계정 소유 트리거 OFF, 에디터 재배포, 폰 도착 확인 등. 넘길 땐 *왜 이건 사용자만 할 수 있는지* 한 줄 이유를 붙인다.
