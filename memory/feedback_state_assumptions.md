코드 작성 전에 내가 깔고 있는 **가정**을 1줄로 명시하고 사용자 확인부터 받는다. 가정이 틀렸으면 코드 전체가 틀어진다.

**Why:** 2026-05-14 *추이 기록* 시트 갱신 작업에서 "운용중/대기"의 의미를 사용자에게 묻지 않고 카테고리(KIS_SKIP)로 임의 매핑해서 운용중 합이 빠진 채 기록됨. 이전에도 _normCode 패딩 버그, MobileAPI.js 옛 23컬럼 가정, mobile API 누락 등이 모두 **검증을 작업 후로 미루고 가정으로 코드부터 짠** 같은 패턴.

가정으로 코드부터 짜면:
1. 잘못된 가정이 코드에 박힘
2. 사용자가 결과 보고 틀린 곳 찾아 알려줘야 함
3. 작업 시간이 사용자 디버깅 시간으로 흘러감

**How to apply:**
- 옛 코드를 v2/신시스템으로 옮길 때 → 데이터 소스·컬럼 인덱스가 같은가? 1줄로 명시 후 확인
- 익숙하지 않은 시트/데이터 구조 다룰 때 → 컬럼 의미·필드 정의가 내 추측대로인가? 1줄로 명시 후 확인
- 외부 클라이언트가 의존하는 함수 작성 시 → 클라이언트가 호출하는 함수명·파라미터·반환 형식 검증 후 작성
- 카테고리/그룹 분류 로직 작성 시 → "이 카테고리는 A 그룹, 저 카테고리는 B 그룹"의 정의가 사용자 의도와 같은가? 1줄로 명시 후 확인

작업 후 "결과 보고 알려주세요"로 끝내지 말 것. 작업 전 "이 가정으로 진행할게요"로 시작할 것.
